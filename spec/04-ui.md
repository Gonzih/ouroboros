# Spec: packages/ui

## Purpose
Observe the system (jobs, logs) and drive it (submit feedback, register MCPs). Evolves via the Ouroboros loop — user feedback triggers code changes to the Vue components directly.

## Stack

- **Vue 3** (Composition API, `<script setup>`) — lightweight, no JSX, template syntax is easy for the meta-agent to modify
- **TypeScript** throughout — `.vue` SFCs with full type safety
- **Vite** — build tool, fast HMR for dev, produces a static bundle for production
- **Pinia** — state management (jobs, logs, MCPs as reactive stores)
- **No UI component library** — custom CSS, dark terminal aesthetic (JetBrains Mono, same palette as cc-agent-ui). Keeps the meta-agent free to evolve the look without fighting a component library.
- **WebSocket** — live updates from server (`ws` library on server side)
- **Node.js Express server** (`server.ts`) — serves the built `dist/` + WebSocket endpoint + REST API
- Default port: **7702**

Rationale: Vue 3 SFCs are compact and readable — a single component file is as easy for the meta-agent to modify as a vanilla HTML file, but with reactivity, types, and proper component structure. Vite build is fast and the output is a static bundle the server can serve. TypeScript catches regressions the meta-agent might introduce.

## Project Structure

```
packages/ui/
  src/
    App.vue              — root, tab navigation
    main.ts              — Vue app mount
    stores/
      jobs.ts            — Pinia store: Job[]
      logs.ts            — Pinia store: LogEntry[], filtered stream
      mcp.ts             — Pinia store: McpConfig[]
      feedback.ts        — Pinia store: FeedbackEvent[], submit action
    components/
      Dashboard.vue      — active jobs, health, quick inputs
      JobList.vue        — full job table, expandable output
      LogStream.vue      — live filtered log view
      FeedbackPanel.vue  — submit feedback, evolution history
      McpRegistry.vue    — list MCPs, register form, test button
    composables/
      useWebSocket.ts    — WebSocket connection, auto-reconnect
      useApi.ts          — typed fetch wrappers for REST endpoints
    types.ts             — re-exports from @ouroboros/core
  server.ts              — Express + WebSocket server
  vite.config.ts
  tsconfig.json
  package.json
```

## Views (tab-based SPA)

### Dashboard
- Active jobs count, recent 5 log lines, meta-agent health (PID or "not running")
- Quick task input: text + backend selector (git/local) + target field + Run button
- Quick feedback input: text + Send button (triggers Ouroboros loop)

### Jobs
- Table: job ID (8 chars), backend, target (truncated), status badge, created, elapsed time
- Click row → inline expand showing last 100 output lines, live-updating via WebSocket if running
- Status badges: pending=grey, running=amber, completed=green, failed=red

### Logs
- Live scrolling feed from `ouro:logs` via WebSocket
- Filter chips: all | meta-agent | worker | gateway | mcp-factory
- Auto-scroll toggle (default on)

### Feedback
- Textarea + Submit button
- Warning banner: "Submitting feedback triggers code changes to Ouroboros. You will receive a diff to approve."
- Recent feedback history: text, timestamp, status (pending / pr-open / approved / rejected)
- Pending evolutions show PR URL + diff summary + Approve/Reject buttons

### MCP Registry
- List: name, scheme, operational badge (green/red), registered timestamp
- Register form: name + connection string + Register button
- Each row has a Test button → calls `POST /api/mcp/test/:name`

## WebSocket Protocol (same as spec/04 previous, unchanged)

Server → client:
```typescript
type WsMessage =
  | { type: 'log'; entry: LogEntry }
  | { type: 'job_update'; job: Partial<Job> }
  | { type: 'job_output'; jobId: string; line: string }
  | { type: 'notify'; text: string }
```

Client → server:
```typescript
type WsCommand =
  | { type: 'subscribe_job'; jobId: string }
  | { type: 'unsubscribe_job'; jobId: string }
```

## REST API (server.ts)

```
POST /api/feedback              body: { text }              → push to ouro:feedback
POST /api/task                  body: { instructions, backend, target } → push to ouro:tasks
POST /api/feedback/:id/approve  → set ouro:feedback:{id}:approval = approved
POST /api/feedback/:id/reject   body: { reason }            → set rejected
POST /api/mcp/register          body: { name, connectionString } → forward to mcp-factory
POST /api/mcp/test/:name        → trigger re-validation in mcp-factory
GET  /api/jobs                  → Job[] from ouro:jobs:*
GET  /api/health                → { redis: bool, metaAgentPid: string|null, jobCount: number }
```

## Authentication

None for v1. Future: OIDC SSO via `OURO_OIDC_ISSUER` env var — suitable for corporate deployment. Single env var switches it on, zero code change needed.
