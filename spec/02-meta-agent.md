# Spec: packages/meta-agent

## Purpose
The always-on coordinator. Runs four concurrent loops driven by Postgres. Singleton — advisory lock ensures only one instance across all platforms.

## Startup Sequence

```
1. Run migrate() — ensure schema + extensions exist
2. pg_try_advisory_lock('ouro:meta-agent') — exit 0 if already held
3. Register SIGTERM/SIGINT handlers → pg_advisory_unlock + exit
4. Crash recovery: UPDATE ouro_jobs SET status='pending', started_at=NULL WHERE status='running'
5. Start four loops concurrently (Promise.all)
6. Log startup: "meta-agent started (PID {pid})"
```

## Cross-Platform Process Management

No launchd, no systemd hardcoded. The meta-agent is a plain Node.js process. How it's kept alive is up to the installer:
- **macOS**: launchd plist (generated by installer)
- **Linux**: systemd unit (generated by installer)
- **Windows**: Task Scheduler or NSSM (generated by installer)
- **Dev**: `node packages/meta-agent/dist/index.js` directly

The installer script (`scripts/install.sh` / `scripts/install.ps1`) generates the appropriate supervisor config for the detected platform.

## Loop 1 — MCP Registry Watch

```
LISTEN 'ouro_notify' events for type='mcp_registered'
  → read new entry from ouro_mcp_registry
  → log "MCP {name} now available"
  → notify gateway to inform user
```

This loop is mostly passive — mcp-factory does the work. Meta-agent reacts to registration events and keeps users informed.

## Loop 2 — Worker Dispatch

```
poll pgmq ouro_tasks every 2 seconds (dequeue, visibility timeout 30min):
  → deserialize: { id, backend, target, instructions, sessionId? }
  → UPDATE ouro_jobs SET status='running', started_at=NOW() WHERE id=$1
  → spawn worker subprocess: node packages/worker/dist/index.js
      env: OURO_TASK=<json>, DATABASE_URL, CLAUDE_CODE_OAUTH_TOKEN, GITHUB_TOKEN
  → stream stdout lines → INSERT INTO ouro_job_output (job_id, line)
  → on exit 0: UPDATE status='completed', ack message
  → on exit !=0: UPDATE status='failed', error=last stderr, nack (requeue once)
  → NOTIFY 'ouro_notify', { type: 'job_complete', jobId, status }
```

Max concurrent workers: 3 (configurable via `OURO_MAX_WORKERS`, default 3).

Worker slot reservation (`activeWorkers.set(id, null)`) happens before the async spawn to avoid a TOCTOU race under concurrent poll ticks.

## Loop 3 — Self-Evolution (Ouroboros Loop)

```
poll pgmq ouro_feedback every 5 seconds (dequeue, visibility timeout 10min):
  → deserialize FeedbackEvent
  → build evolution prompt (see below)
  → spawnSync claude:
      claude --print --dangerously-skip-permissions -p "{prompt}"
      cwd: OURO_REPO_ROOT
      env: includes CLAUDE_CODE_OAUTH_TOKEN
      timeout: 5 minutes
  → check for PR_OPENED:{url} marker in output
  → UPDATE ouro_feedback SET status='pr_open', pr_url=url WHERE id=$1
  → NOTIFY 'ouro_notify', { type: 'evolution_proposed', feedbackId, prUrl, diff }
  → gateway sends user: "Ouroboros updated itself. Here's what changed: [diff]. /approve {id} or /reject {id}"
  → wait for approval (poll ouro_feedback WHERE id=$1 AND status IN ('approved','rejected'))
  → on approved: pnpm build → if success: releaseLock → sleep(2000) → process.exit(0)
  → on rejected: spawnSync claude "close PR {url}" → UPDATE status='rejected', nack
  → supervisor (launchd/systemd/NSSM) respawns meta-agent with new binary
```

**Never restart on build failure** — the running binary is still valid; crashing it would leave the system with no coordinator.

### Evolution Prompt Template

```
You are operating on the Ouroboros codebase at {repoRoot}.

A user submitted this feedback:
"{feedbackText}"

Implement this as a code change:
1. git checkout -b feat/feedback-{id}
2. Make the change
3. Run: pnpm build (verify it compiles without errors)
4. git add -A && git commit -m "feat: {first 60 chars of feedback}"
5. gh pr create --title "feat: {first 60 chars}" --body "{feedbackText}" --base main

When PR is opened, respond with exactly: PR_OPENED:{pr_url}
If build fails, respond with exactly: BUILD_FAILED:{error}
```

## Loop 4 — Watchdog (self-healing)

Runs every 60 seconds. Detects dead worker PIDs and stale service processes.

### Stale job detection

1. Query `ouro_jobs WHERE status='running' AND last_heartbeat < NOW() - 10 min`
2. For each: `isPidAlive(job.pid)` — if dead or heartbeat stale:
   - UPDATE job to `status='pending', pid=NULL, last_heartbeat=NULL, started_at=NULL`
   - Re-enqueue to `ouro_tasks` with `sessionId` preserved (so resumed worker uses `--continue`)
   - Publish `{ type: 'job_requeued', jobId, reason: 'watchdog_dead_pid' }`

The old process is not killed. It may still be writing output. The new worker resumes where the session left off.

### Service restart detection

1. Query `ouro_processes WHERE name IN ('gateway', 'ui')`
2. For each dead PID: unregister from `ouro_processes`, call `state.restartService(name, command, args)`

### PID liveness check

```typescript
function isPidAlive(pid: number): boolean {
  try {
    process.kill(pid, 0)  // signal 0 = existence check, no real signal sent
    return true
  } catch (err) {
    // EPERM: process exists but we can't signal it → still alive
    // ESRCH: process does not exist → dead
    if (err instanceof Error && (err as NodeJS.ErrnoException).code === 'EPERM') {
      return true
    }
    return false
  }
}
```

Cross-platform: `kill(pid, 0)` works on macOS, Linux, and Windows (Node.js normalises it).

## Advisory Lock (Postgres)

```typescript
const acquired = await tryAcquireLock('ouro:meta-agent')
if (!acquired) {
  console.log('meta-agent: another instance is running — exiting')
  process.exit(0)
}

// Lock is held for the lifetime of the Postgres connection
// Process crash = connection closed = lock auto-released
// No TTL needed, no heartbeat needed
```

This is cleaner than Redis NX+TTL: no stale lock on crash, no heartbeat loop.

## v0.2 Vision: Meta-Agent as Persistent Claude Session

In v0.1.0, the meta-agent is a Node.js polling loop that spawns one-shot `claude` subprocesses.

In v0.2, when `@ouroboros/mcp-server` exists, the meta-agent itself becomes a persistent Claude session:

```bash
claude --continue \
  --mcp-config ouroboros-control.json \
  --dangerously-skip-permissions
```

Claude becomes the coordinator instead of Node.js. The polling loops are replaced by Claude reasoning over MCP tool calls:

- `list_jobs()` instead of polling pgmq
- `spawn_worker()` instead of shelling out
- `approve_evolution()` instead of watching ouro_feedback
- `get_logs()` instead of tailing stdout

The Node.js code in v0.1 is the bootstrap that makes v0.2 possible. See `spec/09-mcp-server.md`.
